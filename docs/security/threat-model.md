# Threat model

Status: Checkpoint 4 integration review after the protected session, hosted fail-closed, readiness, and application-root deployment changes. Reassess after any authentication, provider, persistence, protocol, deployment, or data-classification change.

## Scope and protected properties

In scope: the Next.js patient/clinician PWA, API routes, deterministic orchestration, PostgreSQL repository, synthetic fixtures/demo tools, Vercel/Neon deployment shape, ElevenLabs token boundary, VitalLens proxy, GitHub Actions, and operator procedures.

Protect:

- workflow safety and provenance;
- synthetic patient/task/audit integrity and availability;
- server-only database, demo, ElevenLabs, and VitalLens secrets;
- short-lived demo/provider session credentials;
- transient camera/audio and provider payloads;
- protocol/version, quality, idempotency, and audit evidence;
- honest separation of automated fixtures from live/provider/device evidence.

Out of scope as implemented: real patient data, live eMed/EHR connectivity, real clinical identity/tenancy, diagnosis/treatment decisions, and medical-device operation. Introducing any of them invalidates this model.

## Trust boundaries

1. **Browser to HomeRounds API:** all browser input, headers, cookies, state versions, reports, quality claims, and identifiers are untrusted.
2. **Provider to application:** voice transcripts, model proposals, ElevenLabs tokens/events, and VitalLens responses are untrusted until strict validation and deterministic gates run.
3. **Server to PostgreSQL:** repositories enforce schema, transactions, optimistic concurrency, idempotency, append-only audit, and sensitive-payload restrictions; the database/operator account remains privileged.
4. **HomeRounds to providers:** ElevenLabs receives browser audio over its session after a server-issued token; VitalLens receives the approved low-resolution payload through the server proxy. Local finger PPG crosses no network boundary.
5. **Source/CI/deployment:** pull-request content and third-party actions/images are supply-chain inputs. Vercel/Neon/GitHub administrators and environment stores are privileged external controls.

## Abuse cases to keep in release tests

- crafted text asks the model/provider to skip red flags, set urgency, diagnose, or execute a task;
- stale browser writes to a closed/advanced round;
- reused idempotency key with incompatible input and concurrent duplicate clicks;
- malformed/oversized JSON or VitalLens binary metadata/payload;
- provider success with low/unusable quality;
- malicious nested audit keys such as transcript, token, raw frame, or database URL;
- tampered fixture/protocol/version, wrong patient scope, role confusion, and invalid session signature;
- missing PostgreSQL/secret/provider configuration;
- client-submitted passing quality with no trustworthy capture provenance.

## Acceptance boundary

No threat is “solved” merely by this document. Release evidence must include passing automated controls, database/runtime inspection, credential/configuration review, observed provider/network boundaries when enabled, and named waivers. Physical sensor behavior, provider retention, and clinical validity require separate human evidence.
